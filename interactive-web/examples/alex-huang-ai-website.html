Building AI-Assisted Websites
from the Ground Up

A practitioner's guide to embedding LLM capabilities into production web applications — architecture, tradeoffs, and lessons from real deployments.

Author: Alex Huang March 2026 12 min read

Implementation Walkthrough

Step 1 of 5
01

Define the LLM's Role in Your Architecture

Before writing a single API call, decide whether your LLM is an orchestrator, a worker, or a co-processor. This single decision determines your entire reliability profile.

Core Insight
LLMs as orchestrators (deciding what to do) require far more guard-rails than LLMs as workers (executing well-defined sub-tasks). Most production failures come from conflating these two modes.
orchestrator worker co-processor high-risk: orchestrator without fallback

The clearest signal: if your LLM output directly invokes an API or modifies state, it is an orchestrator. Apply the same scrutiny you would to a deployment script running with production credentials.

02

Structure Your Prompt Layer as Infrastructure

Prompts are not strings — they are configuration. Treat them with the same versioning, testing, and review discipline you apply to application code.

// prompts/article-transform.v2.ts
export const ARTICLE_TRANSFORM = {
  version: "2.1.0",
  system: `You are a content intelligence engine...`,
  userTemplate: (article: string) =>
    `Transform the following article:\n\n${article}`,
  temperature: 0.3,
  maxTokens: 4096,
};
Lesson Learned
Prompt drift — where prompts accumulate informal edits without review — was the leading cause of silent regressions in every team we interviewed. Semver your prompts.
03

Design for Latency Budgets, Not Best-Case Performance

LLM API calls have p99 latencies that are 5–20× their median. Build UX and backend timeouts around the p99, not the median.

p50: ~800ms p95: ~4s p99: ~12s

Strategies that work in production:

// 1. Stream tokens to the user immediately
const stream = await client.messages.stream({ ... });
for await (const chunk of stream) {
  res.write(chunk.delta?.text ?? "");
}

// 3. Async with job queue for non-interactive flows
04

Evaluation Before Deployment

You cannot A/B test your way out of a bad foundation. Evals must be in place before you ship, not after you observe failures.

Minimum Viable Eval Suite
Three eval categories: correctness (does output match ground truth?), format (does output parse?), safety (does output refuse jailbreaks?). Start here, expand from incidents.
correctness eval format eval safety eval regression eval (add after first incident)
05

Observability: Log What the Model Saw, Not Just What It Returned

Standard application logging captures outputs. AI-assisted systems require capturing the full context: prompt, model version, temperature, and the full completion — not just the parsed result.

// Log the full LLM interaction, not just parsed results
logger.info("llm_call", {
  prompt_hash: hash(finalPrompt),
  model: "claude-sonnet-4-6",
  temperature: 0.3,
  input_tokens: usage.input_tokens,
  output_tokens: usage.output_tokens,
  latency_ms: Date.now() - start,
  // Store full completion in cold storage, reference by ID
  completion_id: await store(completion),
});
Why This Matters
When a user reports "the AI said something wrong," you need to replay exactly what the model received. Without full prompt logging, post-mortems are guesswork.

Key Takeaways

Building AI-assisted websites is an architectural discipline, not a feature flag. The teams that succeed treat LLM integration with the same rigor they apply to databases: schema the inputs, version the configuration, test before shipping, and instrument everything.
